Original problemปัญหาเดิม

DevStarter v5.7.0 shipped an Obsidian vault but had no project-level note type. Team members joining a project had no vault entry point — they had to read CLAUDE.md manually. Five major workflows had zero vault coverage.DevStarter v5.7.0 มี vault แต่ไม่มีประเภทโน้ตระดับโปรเจกต์ สมาชิกทีมใหม่ไม่มีจุดเริ่มต้นใน vault ต้องอ่าน CLAUDE.md เอง workflow หลัก 5 รายการไม่มี vault coverage

How it was resolvedแก้ไขอย่างไร

Added a new project-snapshot Obsidian note template and wired optional emit points into three commands: /devstarter-new (Gate 0 post-scaffold), /devstarter-release (Phase 9.5 alongside launch brief), and vault recall into /devstarter-onboard (session start). All emit points are opt-in via AskUserQuestion and silently skip when obsidian.enabled: false.เพิ่ม template โน้ต project-snapshot ใหม่และเชื่อมต่อ optional emit point เข้ากับสามคำสั่ง: /devstarter-new (Gate 0 หลัง scaffold), /devstarter-release (Phase 9.5 พร้อม launch brief), และ recall เข้า /devstarter-onboard (ต้น session) จุด emit ทั้งหมดเป็น opt-in และข้ามโดยอัตโนมัติเมื่อ obsidian.enabled: false

Key Implementation Decisionsการตัดสินใจหลักในการพัฒนา

  • Gate 0 vs Gate 5 for /devstarter-new: Chose Gate 0 (post-scaffold setup) over Gate 5 (post-development) — captures initial project config before any code is written, which is more useful for onboarding context than end-of-sprint state.Gate 0 vs Gate 5 สำหรับ /devstarter-new: เลือก Gate 0 (หลัง scaffold) แทน Gate 5 (หลังพัฒนา) เพื่อบันทึก config โปรเจกต์เริ่มต้นก่อนเขียนโค้ด ซึ่งมีประโยชน์สำหรับ onboarding มากกว่า
  • Stacking over overwriting: Each release emits a new unique file (Rule 3). This gives a project evolution timeline in the vault and avoids concurrent-write corruption on network shares.Stack แทน overwrite: แต่ละ release emit ไฟล์ใหม่ที่ไม่ซ้ำกัน (Rule 3) ทำให้ vault มี timeline วิวัฒนาการโปรเจกต์และหลีกเลี่ยงปัญหาการเขียนพร้อมกันบน network share
  • Opt-in gating: All emit points use AskUserQuestion rather than auto-emitting. This respects teams that haven't set up Obsidian, avoids vault noise from dry-run sessions, and matches the existing vault emit pattern (E5 confirmation).Opt-in gate: จุด emit ทั้งหมดใช้ AskUserQuestion แทนการ emit อัตโนมัติ เพื่อเคารพทีมที่ยังไม่ตั้งค่า Obsidian หลีกเลี่ยง vault noise และตรงกับ pattern emit เดิม

Deviations from Planการเปลี่ยนจากแผน

Minor deviation: Gate 0 vs "Phase 5 post-scaffold"เบี่ยงเบนเล็กน้อย: Gate 0 แทน "Phase 5 post-scaffold"

The plan.html refers to "/devstarter-new Phase 5 post-scaffold gate" but the actual devstarter-starter.md uses Gates (0–5), not Phases. The implementation wires into Gate 0 (the setup phase) rather than a non-existent "Phase 5". This is a terminology mismatch in the plan — the intent was identical. BRD REQ-OBS-002 should be updated to read "Gate 0 post-scaffold" at next BRD review.plan.html อ้างถึง "Phase 5 post-scaffold gate" แต่ devstarter-starter.md จริงใช้ Gates (0-5) ไม่ใช่ Phases การดำเนินงาน wire ไปที่ Gate 0 (ขั้นตอน setup) แทน "Phase 5" ที่ไม่มีอยู่จริง เป็นความไม่ตรงกันของ terminology ใน plan ควรอัปเดต BRD REQ-OBS-002 ในการรีวิว BRD ครั้งถัดไป
