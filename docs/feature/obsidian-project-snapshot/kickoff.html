Overviewภาพรวม

📝 What you are being asked to approve

We are adding a project-snapshot note type to DevStarter's Obsidian Knowledge Vault. When a new project is created or a release is shipped, DevStarter will optionally capture a snapshot note — tech stack, architecture decisions, constraints, repo URL — that team members can browse in Obsidian to orient themselves. This closes the gap where new developers joining a project have no vault entry point for project-level context.เราจะเพิ่มประเภทโน้ตใหม่ชื่อ project-snapshot เข้าสู่ Obsidian Knowledge Vault ของ DevStarter เมื่อสร้างโปรเจกต์ใหม่หรือปล่อยเวอร์ชันใหม่ DevStarter จะเสนอให้บันทึกโน้ต snapshot ที่ระบุ tech stack การตัดสินใจสถาปัตยกรรม ข้อจำกัด และ URL ของ repo เพื่อให้สมาชิกทีมเปิดดูใน Obsidian และเริ่มต้นทำความเข้าใจโปรเจกต์ได้ทันที

What We Will Buildสิ่งที่เราจะสร้าง

👤 For the requester — please confirm this matches your intentสำหรับผู้ร้องขอ — โปรดยืนยันว่าตรงกับความต้องการ

We will add a new Obsidian note type called project-snapshot. This is a single Markdown template that captures a project's tech stack, key architecture decisions, constraints, and repository URL in a structured, searchable format. Three DevStarter commands will integrate with it: /devstarter-new will offer to emit a snapshot after project scaffold; /devstarter-release will emit a versioned snapshot at each release milestone; and /devstarter-onboard will automatically grep the vault and surface the snapshot as the first orientation note for new team members.เราจะเพิ่มประเภทโน้ต Obsidian ใหม่ชื่อ project-snapshot ซึ่งเป็น template Markdown ที่บันทึก tech stack ของโปรเจกต์ การตัดสินใจสถาปัตยกรรมหลัก ข้อจำกัด และ URL ของ repo ในรูปแบบที่มีโครงสร้างและค้นหาได้ คำสั่งสามคำสั่งของ DevStarter จะรองรับสิ่งนี้ ได้แก่ /devstarter-new จะเสนอให้บันทึก snapshot หลังสร้างโปรเจกต์ /devstarter-release จะบันทึก snapshot เวอร์ชันใหม่ทุกครั้งที่ปล่อยเวอร์ชัน และ /devstarter-onboard จะ grep vault โดยอัตโนมัติและแสดง snapshot เป็นโน้ตปฐมนิเทศสำหรับสมาชิกใหม่

User StoryUser Story

As a developer joining a new project, I want to open the knowledge vault and immediately find a project-snapshot note showing the tech stack, key decisions, and constraints — so that I can orient myself without interrupting the team.ในฐานะนักพัฒนาที่เข้าร่วมโปรเจกต์ใหม่ ฉันต้องการเปิด knowledge vault แล้วพบโน้ต project-snapshot ที่แสดง tech stack การตัดสินใจหลัก และข้อจำกัดทันที — เพื่อให้ฉันปรับตัวได้เองโดยไม่ต้องรบกวนทีม

Acceptance Criteria — done whenเกณฑ์การยอมรับ — เสร็จเมื่อ

  • Given obsidian.enabled: true and vault_path set, When /devstarter-new completes Phase 5, Then Claude asks "Save a project-snapshot note?" and, if yes, emits a sanitized note with type: project-snapshot, stack, architecture-pattern, key-decisions, constraints, repo-url.เมื่อ obsidian.enabled: true และตั้งค่า vault_path แล้ว เมื่อ /devstarter-new เสร็จ Phase 5 Claude จะถามว่า "บันทึก project-snapshot หรือไม่" และหากตอบรับ จะ emit โน้ตที่ผ่านการ sanitize พร้อม type: project-snapshot, stack, architecture-pattern, key-decisions, constraints, repo-url
  • Given a project-snapshot note exists in the vault, When /devstarter-onboard runs, Then it greps the vault for type: project-snapshot matching the current project and surfaces the note as the orientation entry before onboarding steps begin.เมื่อมีโน้ต project-snapshot ใน vault อยู่แล้ว เมื่อ /devstarter-onboard ทำงาน มันจะ grep vault หา type: project-snapshot ที่ตรงกับโปรเจกต์ปัจจุบัน และแสดงโน้ตนั้นเป็นข้อมูลปฐมนิเทศก่อนเริ่มขั้นตอน onboarding
  • Given obsidian.enabled: true, When /devstarter-release Phase 9.5 runs, Then a new versioned project-snapshot note (unique filename with version key) is offered for emit, building a release timeline in the vault.เมื่อ obsidian.enabled: true เมื่อ /devstarter-release Phase 9.5 ทำงาน จะเสนอการ emit โน้ต project-snapshot เวอร์ชันใหม่ (ชื่อไฟล์ไม่ซ้ำพร้อม key เวอร์ชัน) เพื่อสร้างประวัติการ release ใน vault
  • All workflows still work correctly when obsidian.enabled: false — emit steps silently skip.ทุก workflow ยังทำงานได้ปกติเมื่อ obsidian.enabled: false — ขั้นตอน emit จะข้ามโดยอัตโนมัติ

Management Summaryสรุปสำหรับผู้บริหาร

📊 For management — business rationale, no technical detailสำหรับผู้บริหาร — เหตุผลเชิงธุรกิจ ไม่มีรายละเอียดทางเทคนิค

Why now — the business needทำไมต้องทำตอนนี้ — ความจำเป็นในปัจจุบัน

DevStarter v5.7.0 introduced a shared knowledge vault for capturing bugs and root causes. However, there is no "project overview" record — new team members and developers switching projects still have no vault entry point to understand a project at a glance. This feature fills that gap by capturing project context automatically at project creation and each release.DevStarter v5.7.0 เพิ่ม knowledge vault สำหรับบันทึก bug และต้นตอของปัญหา แต่ยังไม่มี "ภาพรวมโปรเจกต์" สมาชิกทีมใหม่และนักพัฒนาที่สลับโปรเจกต์ยังไม่มีจุดเริ่มต้นใน vault ฟีเจอร์นี้จะแก้ปัญหาดังกล่าวด้วยการบันทึก context โปรเจกต์อัตโนมัติเมื่อสร้างโปรเจกต์ใหม่และทุกครั้งที่ปล่อยเวอร์ชัน

Who benefitsใครได้ประโยชน์

All developers using DevStarter — especially those joining existing projects mid-stream, or solo developers who want a browsable project history. Teams using Obsidian will see an immediate improvement in onboarding speed.นักพัฒนาทุกคนที่ใช้ DevStarter โดยเฉพาะผู้ที่เข้าร่วมโปรเจกต์ที่ดำเนินอยู่แล้ว หรือนักพัฒนาเดี่ยวที่ต้องการประวัติโปรเจกต์ที่เรียกดูได้ ทีมที่ใช้ Obsidian จะเห็นการปรับปรุงความเร็วในการ onboarding ทันที

Cost of not doing thisผลกระทบหากไม่ดำเนินการ

The knowledge vault remains a bug-and-fix archive with no project-level context. Team onboarding continues to rely on reading CLAUDE.md manually. The vault's value as a team knowledge base is limited to post-incident records only.knowledge vault จะยังคงเป็นที่เก็บบันทึก bug เท่านั้น โดยไม่มี context ระดับโปรเจกต์ การ onboarding ทีมยังต้องพึ่งการอ่าน CLAUDE.md ด้วยตนเอง ประโยชน์ของ vault ในฐานะฐานความรู้ทีมจะจำกัดอยู่แค่บันทึกหลังเหตุการณ์เท่านั้น
