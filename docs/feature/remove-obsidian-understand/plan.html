Reason for Removalเหตุผลในการนำออก

Root Problemต้นตอของปัญหา

Both feature sets were added to extend DevStarter's scope beyond its core purpose. The Obsidian integration (v5.7.0–v6.0.0) introduced 9 integration points across the SDLC runbooks, creating ongoing maintenance burden. The Understand-Anything wrappers (v5.6.0) added 8 commands that delegate to an external plugin dependency. Neither feature set aligns with the toolkit's core mission of structured SDLC automation. ฟีเจอร์ทั้งสองชุดถูกเพิ่มเพื่อขยายขอบเขตของ DevStarter เกินกว่าจุดประสงค์หลัก การผสานรวม Obsidian (v5.7.0–v6.0.0) สร้างจุดเชื่อมต่อ 9 จุดใน SDLC runbook ก่อให้เกิดภาระการบำรุงรักษาต่อเนื่อง wrapper ของ Understand-Anything (v5.6.0) เพิ่ม command 8 รายการที่ส่งต่อไปยัง plugin ภายนอก ฟีเจอร์ทั้งสองชุดไม่สอดคล้องกับพันธกิจหลักของ toolkit ในการทำให้ SDLC เป็นอัตโนมัติอย่างมีโครงสร้าง

Business Requirementความต้องการทางธุรกิจ

Project owner explicitly requested removal of both feature sets. Simplify the toolkit to its core SDLC automation commands, reduce maintenance surface, and eliminate the external plugin dependency for Understand-Anything. เจ้าของโปรเจกต์ร้องขอให้นำฟีเจอร์ทั้งสองชุดออก เพื่อลดความซับซ้อนของ toolkit ให้เหลือเพียง command อัตโนมัติ SDLC หลัก ลด surface area การบำรุงรักษา และขจัด dependency ต่อ plugin ภายนอกสำหรับ Understand-Anything

User Storyเรื่องราวของผู้ใช้

As a DevStarter maintainer,
I want to remove the Obsidian vault and Understand-Anything features entirely,
So that the toolkit stays focused on core SDLC automation with no external plugin dependencies.

Removal Approachแนวทางการนำออก

Approachแนวทาง

Clean delete: remove all primary files first (skills dirs, SDLC runbooks, templates, ADRs, feature docs, runtime artifacts), then edit each reference file to strip vault/understand blocks. The approach is ordered upstream-to-downstream to avoid editing a file that references a file we haven't deleted yet. ลบแบบสะอาด: นำไฟล์หลักทั้งหมดออกก่อน (skills dir, SDLC runbook, template, ADR, feature doc, runtime artifact) จากนั้นแก้ไขแต่ละไฟล์อ้างอิงเพื่อลบบล็อก vault/understand ลำดับเป็นไปจาก upstream ไป downstream เพื่อหลีกเลี่ยงการแก้ไขไฟล์ที่อ้างอิงไฟล์ที่ยังไม่ได้ลบ

Why This Approachเหตุผลที่เลือกแนวทางนี้

Deprecation-first is unnecessary — these features have no external consumers (DevStarter is a local toolkit installed per-user). A clean delete produces a cleaner result with no dead code, no deprecation notices cluttering the runbooks, and a smaller diff that is easier to review and revert if needed. การ deprecate ก่อนไม่จำเป็น เนื่องจากฟีเจอร์เหล่านี้ไม่มีผู้ใช้ภายนอก (DevStarter เป็น local toolkit ที่ install ต่อผู้ใช้) การลบแบบสะอาดให้ผลลัพธ์ที่ดีกว่า ไม่มีโค้ดที่ตายแล้ว ไม่มีประกาศ deprecation ที่รบกวน runbook และมี diff ที่เล็กกว่าซึ่งง่ายต่อการ review และ revert หากจำเป็น

Alternatives Consideredทางเลือกอื่นที่พิจารณา

OptionReason Rejected
Deprecate first, delete laterDeprecate ก่อน ลบทีหลังNo external consumers. Adds noise without value. Owner wants clean removal now.ไม่มีผู้ใช้ภายนอก เพิ่ม noise โดยไม่มีประโยชน์ เจ้าของต้องการลบแบบสะอาดทันที
Move to an optional --profile flagย้ายไปเป็น optional --profile flagIncreases install complexity. Owner decision is to remove entirely.เพิ่มความซับซ้อนในการติดตั้ง เจ้าของตัดสินใจนำออกทั้งหมด
