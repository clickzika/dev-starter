Problem & Business Requirementปัญหา & ความต้องการ

v5.8.0 closed the team-overview gap. v5.9.0 closes the knowledge-loop gap: vault notes from code reviews, audit findings, and retrospectives are not captured automatically, so teams continue to repeat known problems. And when starting a new feature, the vault's technique notes are not surfaced, so developers miss prior art.v5.8.0 ปิดช่องว่าง team-overview แล้ว v5.9.0 ปิดช่องว่าง knowledge-loop: vault notes จาก code review, audit, และ retro ไม่ถูกบันทึกอัตโนมัติ ทำให้ทีมยังทำผิดซ้ำ และเมื่อเริ่ม feature ใหม่ technique notes ใน vault ไม่ถูกแสดง ทำให้นักพัฒนาพลาด prior art

User storiesUser stories

  • As a developer starting a new feature, I want vault technique notes surfaced before design begins — so I apply known patterns.ในฐานะนักพัฒนาเริ่ม feature ใหม่ ต้องการให้ vault technique notes ปรากฏก่อนเริ่มออกแบบ
  • As a code reviewer, I want to capture recurring findings to the vault — so patterns reach other projects.ในฐานะ code reviewer ต้องการบันทึก recurring findings ลง vault เพื่อแชร์ข้ามโปรเจกต์
  • As an auditor, I want critical findings captured as rca notes — so audit patterns are searchable.ในฐานะ auditor ต้องการบันทึก critical findings เป็น rca notes ที่ค้นหาได้
  • As a retrospective facilitator, I want process lessons captured — so teams learn from each other's sprints.ในฐานะ retrospective facilitator ต้องการบันทึก process lessons เพื่อให้ทีมเรียนรู้จากกัน

Solution Approachแนวทางแก้ปัญหา

Follow the exact pattern established in v5.8.0: minimal additive blocks that check obsidian.enabled first, use the Vault Recall Procedure (recall) or Vault Emit Procedure E1-E5 (emit), and silently skip when disabled. No new templates — reuse type: technique and type: rca already defined in sdlc/devstarter-knowledge.md.ตาม pattern ที่สร้างใน v5.8.0: additive block ขนาดเล็กที่ตรวจสอบ obsidian.enabled ก่อน ใช้ Vault Recall Procedure หรือ Vault Emit Procedure E1-E5 และข้ามโดยไม่มีข้อความเมื่อปิด ไม่มี template ใหม่ ใช้ซ้ำ type: technique และ type: rca

  1. Task 1 — Vault recall in /devstarter-change-add.mdTask 1 — Vault recall ใน /devstarter-change-add.md
    • Insert before ## A-PHASE 2 — Impact Analysis headerแทรกก่อน header ## A-PHASE 2 — Impact Analysis
    • Check obsidian.enabled; silently skip if falseตรวจสอบ obsidian.enabled; ข้ามโดยไม่มีข้อความถ้าปิด
    • Run Vault Recall Procedure: derive 2–4 keywords from feature name + topic (A-Q1/A-Q3); grep vault subdir; surface top 1–3 technique matches as "Prior patterns (vault):" blockรัน Vault Recall Procedure: ดึง 2–4 keyword จากชื่อ feature + topic; grep vault; แสดง top 1–3 technique matches
    • If no matches: one line "No matching vault notes found." then proceed to A-PHASE 2ถ้าไม่พบ: แสดงหนึ่งบรรทัดแล้วดำเนินต่อ
  2. Task 2 — Optional emit in /devstarter-review.mdTask 2 — Optional emit ใน /devstarter-review.md
    • Append after the "explain finding" block (end of Phase 4)เพิ่มหลัง "explain finding" block (ท้าย Phase 4)
    • Check obsidian.enabled; silently skip if false

  3. Task 3 — Optional emit in /devstarter-audit.mdTask 3 — Optional emit ใน /devstarter-audit.md

    • Filter for 🔴 Critical and 🟠 High findings from audit report; AskUserQuestion "Save critical/high audit findings to vault?" → if yes, emit each as type: rca, root_cause_category: audit-gap, symptom: finding description, source: docs/audit-report.htmlกรองหา 🔴 Critical และ 🟠 High findings; ถาม "บันทึก audit findings ลง vault?" → ถ้าใช่ emit แต่ละ finding type: rca, root_cause_category: audit-gap
    • If Q3 = report only (no fixes were run): move emit to after Gate 1 audit report approval insteadถ้า Q3 = report only: ย้าย emit ไปหลัง Gate 1 แทน
  4. Task 4 — Optional emit in /devstarter-retrospective.mdTask 4 — Optional emit ใน /devstarter-retrospective.md
    • Append after Phase 4 (GitHub issues + Notion tasks created), before Phase 5 (Update CLAUDE.md)เพิ่มหลัง Phase 4 (สร้าง issues + tasks แล้ว) ก่อน Phase 5
    • Check obsidian.enabled; silently skip if false
    • AskUserQuestion "Save retro lessons to vault?" → if yes: collect Q3 answers (What would you do differently?) + top action items → emit as type: technique, topic: process, one note per significant lessonถาม "บันทึก retro lessons ลง vault?" → ถ้าใช่: รวบรวมคำตอบ Q3 + action items หลัก → emit type: technique, topic: process
