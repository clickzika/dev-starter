Problem / Requirement

Root Problem

All DevStarter HTML documents (plan, summary, mgmt-brief, document-template) are English-only and have no PDF export capability. Thai-speaking stakeholders cannot read these documents in their preferred language, and documents cannot be easily archived or shared outside a browser.

Business Requirement

All DevStarter document templates must support full bilingual content (English and Thai) with a language toggle button, and must provide PDF export via a browser print button. Agents must generate bilingual content (both languages) when creating new documents. Existing generated docs must be regenerated with bilingual content.

User Story

As a DevStarter user working with Thai-speaking stakeholders,
I want to view all DevStarter documents in Thai and export them as PDF,
So that I can share professional bilingual documentation with stakeholders and create offline-ready PDF archives.

Acceptance Criteria

  • Language toggle button (EN ↔ TH) appears in the topbar of all 4 document templates
  • Clicking the toggle switches ALL text in the document between English and Thai
  • Language preference persists across page reloads via localStorage
  • "Export PDF" button appears in the topbar of all 4 document templates
  • Clicking Export PDF opens the browser print dialog; resulting PDF has no sidebar, full-width content, clean layout
  • Thai text renders correctly using Sarabun font (loaded from Google Fonts CDN)
  • Agents generate bilingual content using <span class="lang-en"> / <span class="lang-th"> span pairs for every paragraph, heading, and bullet
  • Bilingual Content Rule documented in agents/shared/devstarter-agent-base.md with format example
  • All 3 existing docs for branch-guard-hook (plan/summary/mgmt-brief) regenerated with bilingual content
3

Proposed Solution

Approach

Add Google Fonts Sarabun to <head> of all templates. Add CSS for .lang-en / .lang-th visibility classes. Add toggleLang() JavaScript function with localStorage persistence. Add a language toggle button in the topbar (next to the existing theme toggle). Enhance @media print CSS to fully hide sidebar and make content full-width. Add "Export PDF" button in topbar calling window.print(). Update agent instructions to require bilingual span pairs in all generated content. Regenerate existing docs with bilingual content.

Why This Approach

Zero external runtime dependencies beyond Google Fonts CDN. Fully consistent with the existing dark/light theme toggle UX pattern. Works at file:// protocol — no web server needed. window.print() produces searchable, text-selectable PDFs (unlike html2canvas screenshot approach). Self-contained HTML files remain portable and archive-safe.

Alternatives Considered

OptionReason Rejected
Separate .th.html files per documentDoubles file count; EN and TH can drift out of sync; docs portal needs dual entries
External JSON i18n + fetch()Fails at file:// protocol; requires web server; adds complexity
html2pdf.js (CDN)~2MB CDN dependency; complex sidebar layout renders poorly; requires internet
jsPDF + html2canvasText is rasterized (not searchable); very slow; large library
