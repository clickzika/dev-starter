Problem / Requirement

Root Problem

Branch Guard exists as text rules only (Rule 9 in devstarter-change.md, Branch Guard section in agents/shared/devstarter-agent-base.md). Claude can still edit files on protected branches (develop, main, uat, master) if the rule is overlooked in autopilot mode, resume flows, or edge cases. There is no technical enforcement — no Claude Code hook intercepts Edit/Write tool calls to verify the current branch before allowing the edit.

Business Requirement

Every Edit/Write tool call must be technically blocked when Claude is on a protected branch (develop, main, master, uat, dev). The block message must instruct Claude to create the correct branch type (feature/[slug] or fix/[slug]) before proceeding. Non-git directories must pass through without false blocks.

User Story

As a developer using DevStarter,
I want to have Claude automatically blocked from editing on develop/main/uat with a clear error and branch creation instruction,
So that no accidental direct commits to protected branches occur — even in autopilot or resume flows.

Acceptance Criteria

  • Given Claude is about to edit any file while on develop, main, master, uat, or dev — the Edit/Write tool call is blocked with an instruction to create a work branch first
  • Given Claude is on feature/my-feature or fix/my-fix — Edit/Write passes through normally, no block
  • Given the working directory is not a git repository — Edit/Write passes through normally (no false block)
  • All existing PostToolUse hooks (accumulator, config-sync) continue to function without change
  • Hook is registered in templates/hooks/hooks.json and installed via install.sh --hooks
3

Proposed Solution

Approach

Add a PreToolUse hook (scripts/hooks/pre-edit-branch-guard.js) that intercepts every Edit and Write tool call. The hook runs git branch --show-current, checks the result against the protected-branch list, and outputs a JSON block decision ({"decision":"block","reason":"..."}) if on a protected branch. If git is unavailable or the directory is not a git repo, the hook passes through silently. The existing text rules (Rule 9, agent-base Branch Guard) stay as primary instruction; the hook is the technical safety net.

Why This Approach

Defense in depth: text rules explain WHY (educates Claude), the hook enforces HOW (hard technical block that cannot be bypassed by model drift, autopilot, or resume flows). The hook message is educational — it names the current branch and tells Claude exactly what branch to create — so Claude recovers correctly rather than just retrying. The implementation follows the exact same Node.js stdin/stdout pass-through pattern as the existing stop-check-console-log.js hook (~60 lines).

Alternatives Considered

OptionReason Rejected
Text rules only (current state)No technical safety net — model can drift or be overridden in autopilot/resume flows
PostToolUse hookRuns AFTER the edit — too late to prevent the write to a protected branch
Git pre-commit hookOnly fires on commit, not on individual file writes — doesn't prevent Claude from writing changes to the working tree on a protected branch

Implementation Steps

  1. Create feature/branch-guard-hook branch from develop
  2. Create scripts/hooks/pre-edit-branch-guard.js:
    — Read stdin (PreToolUse JSON) and pass through to stdout unchanged
    — Run execFileSync('git', ['branch', '--show-current'])
    — If result is in ['develop','main','master','uat','dev']: write block decision JSON to stdout and exit 0
    — If not in git repo (exception): exit 0 (pass through)
    — Block message: names current branch + instructs feature/[slug] or fix/[slug] creation
  3. Update templates/hooks/hooks.json: add PreToolUse array with "matcher": "Edit|Write" pointing to pre-edit-branch-guard.js
  4. Update agents/shared/devstarter-agent-base.md Branch Guard section: add one-line note about hook
  5. Update sdlc/devstarter-change.md Rule 9: add one-line note about hook
  6. Copy all modified files to ~/.claude/
  7. Test: manually trigger Edit on develop → confirm block; switch to feature branch → confirm pass-through
  8. Commit and open PR to develop

Dependencies

  • Node.js 18+ (already required by hooks system)
  • git in PATH (already required by DevStarter)
  • scripts/install-hooks.js (existing — no changes needed)

Testing Approach

Manual functional test: (1) confirm git branch --show-current returns develop, attempt an Edit tool call, verify Claude receives the block decision and creates a work branch instead; (2) checkout feature/test-branch, attempt an Edit tool call, verify it passes through normally; (3) run in a non-git directory, verify pass-through with no error.
