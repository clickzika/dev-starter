Problem / Requirementปัญหา / ความต้องการ

Root Problemปัญหาหลัก

Branch Guard exists as text rules only (Rule 9 in devstarter-change.md, Branch Guard section in agents/shared/devstarter-agent-base.md). Claude can still edit files on protected branches (develop, main, uat, master) if the rule is overlooked in autopilot mode, resume flows, or edge cases. There is no technical enforcement — no Claude Code hook intercepts Edit/Write tool calls to verify the current branch before allowing the edit.Branch Guard ปัจจุบันมีเพียงกฎข้อความเท่านั้น (Rule 9 ใน devstarter-change.md และส่วน Branch Guard ใน agents/shared/devstarter-agent-base.md) Claude ยังสามารถแก้ไขไฟล์บน protected branch (develop, main, uat, master) ได้หากมีการมองข้ามกฎในโหมด autopilot, resume flow หรือกรณีพิเศษ ไม่มีการบังคับใช้ทางเทคนิค — ไม่มี Claude Code hook ที่สกัดกั้นการเรียก tool Edit/Write เพื่อตรวจสอบ branch ปัจจุบันก่อนอนุญาตให้แก้ไข

Business Requirementความต้องการทางธุรกิจ

Every Edit/Write tool call must be technically blocked when Claude is on a protected branch (develop, main, master, uat, dev). The block message must instruct Claude to create the correct branch type (feature/[slug] or fix/[slug]) before proceeding. Non-git directories must pass through without false blocks.การเรียก tool Edit/Write ทุกครั้งต้องถูกบล็อกทางเทคนิคเมื่อ Claude อยู่บน protected branch (develop, main, master, uat, dev) ข้อความบล็อกต้องสั่งให้ Claude สร้าง branch ประเภทที่ถูกต้อง (feature/[slug] หรือ fix/[slug]) ก่อนดำเนินการ ไดเรกทอรีที่ไม่ใช่ git ต้องผ่านได้โดยไม่มีการบล็อกเท็จ

User Storyเรื่องราวผู้ใช้

As aในฐานะ developer using DevStarter,นักพัฒนาที่ใช้ DevStarter,
I want toฉันต้องการให้ have Claude automatically blocked from editing on develop/main/uat with a clear error and branch creation instruction,Claude ถูกบล็อกอัตโนมัติจากการแก้ไขบน develop/main/uat พร้อมข้อผิดพลาดที่ชัดเจนและคำแนะนำการสร้าง branch,
So thatเพื่อให้ no accidental direct commits to protected branches occur — even in autopilot or resume flows.ไม่มีการ commit โดยตรงไปยัง protected branch โดยไม่ตั้งใจ แม้แต่ใน autopilot หรือ resume flow

Acceptance Criteriaเกณฑ์การยอมรับ

  • Given Claude is about to edit any file while on develop, main, master, uat, or dev — the Edit/Write tool call is blocked with an instruction to create a work branch firstเมื่อ Claude กำลังจะแก้ไขไฟล์ใดๆ ขณะอยู่บน develop, main, master, uat, หรือ dev — การเรียก tool Edit/Write จะถูกบล็อกพร้อมคำแนะนำให้สร้าง work branch ก่อน
  • Given Claude is on feature/my-feature or fix/my-fix — Edit/Write passes through normally, no blockเมื่อ Claude อยู่บน feature/my-feature หรือ fix/my-fix — Edit/Write ผ่านได้ตามปกติ ไม่มีการบล็อก
  • Given the working directory is not a git repository — Edit/Write passes through normally (no false block)เมื่อไดเรกทอรีทำงานไม่ใช่ git repository — Edit/Write ผ่านได้ตามปกติ (ไม่มีการบล็อกเท็จ)
  • All existing PostToolUse hooks (accumulator, config-sync) continue to function without changePostToolUse hooks ที่มีอยู่ทั้งหมด (accumulator, config-sync) ยังทำงานได้โดยไม่มีการเปลี่ยนแปลง
  • Hook is registered in templates/hooks/hooks.json and installed via install.sh --hooksHook ถูกลงทะเบียนใน templates/hooks/hooks.json และติดตั้งผ่าน install.sh --hooks

Proposed Solutionวิธีแก้ปัญหาที่เสนอ

Approachแนวทาง

Add a PreToolUse hook (scripts/hooks/pre-edit-branch-guard.js) that intercepts every Edit and Write tool call. The hook runs git branch --show-current, checks the result against the protected-branch list, and outputs a JSON block decision ({"decision":"block","reason":"..."}) if on a protected branch. If git is unavailable or the directory is not a git repo, the hook passes through silently. The existing text rules (Rule 9, agent-base Branch Guard) stay as primary instruction; the hook is the technical safety net.เพิ่ม PreToolUse hook (scripts/hooks/pre-edit-branch-guard.js) ที่สกัดกั้นการเรียก tool Edit และ Write ทุกครั้ง Hook จะรัน git branch --show-current, ตรวจสอบผลลัพธ์กับรายการ protected branch และส่งออกคำตัดสินบล็อก JSON ({"decision":"block","reason":"..."}) หากอยู่บน protected branch ถ้า git ไม่พร้อมใช้งานหรือไดเรกทอรีไม่ใช่ git repo hook จะผ่านโดยไม่ส่งเสียง กฎข้อความที่มีอยู่ (Rule 9, agent-base Branch Guard) ยังคงเป็นคำแนะนำหลัก โดย hook เป็นตาข่ายความปลอดภัยทางเทคนิค

Why This Approachเหตุผลที่เลือกแนวทางนี้

Defense in depth: text rules explain WHY (educates Claude), the hook enforces HOW (hard technical block that cannot be bypassed by model drift, autopilot, or resume flows). The hook message is educational — it names the current branch and tells Claude exactly what branch to create — so Claude recovers correctly rather than just retrying. The implementation follows the exact same Node.js stdin/stdout pass-through pattern as the existing stop-check-console-log.js hook (~60 lines).การป้องกันเชิงลึก: กฎข้อความอธิบายว่า WHY (สั่งสอน Claude), hook บังคับใช้ว่า HOW (การบล็อกทางเทคนิคแบบแข็งที่ไม่สามารถหลีกเลี่ยงได้โดย model drift, autopilot หรือ resume flow) ข้อความ hook เป็นการสอน — มันระบุชื่อ branch ปัจจุบันและบอก Claude ว่าต้องสร้าง branch อะไร — เพื่อให้ Claude กู้คืนได้อย่างถูกต้องแทนที่จะลองใหม่ การพัฒนาตามรูปแบบ Node.js stdin/stdout pass-through เดียวกันกับ hook stop-check-console-log.js ที่มีอยู่ (~60 บรรทัด)

Implementation Stepsขั้นตอนการพัฒนา

  1. Create feature/branch-guard-hook branch from developสร้าง branch feature/branch-guard-hook จาก develop
  2. Create scripts/hooks/pre-edit-branch-guard.js:
    — Read stdin (PreToolUse JSON) and pass through to stdout unchanged
    — Run execFileSync('git', ['branch', '--show-current'])
    — If result is in ['develop','main','master','uat','dev']: write block decision JSON to stdout and exit 0
    — If not in git repo (exception): exit 0 (pass through)
    — Block message: names current branch + instructs feature/[slug] or fix/[slug] creation     สร้าง scripts/hooks/pre-edit-branch-guard.js:
    — อ่าน stdin (PreToolUse JSON) และส่งผ่านไปยัง stdout โดยไม่เปลี่ยนแปลง
    — รัน execFileSync('git', ['branch', '--show-current'])
    — หากผลลัพธ์อยู่ใน ['develop','main','master','uat','dev']: เขียนคำตัดสินบล็อก JSON ไปยัง stdout และออกด้วย 0
    — หากไม่ได้อยู่ใน git repo (exception): ออกด้วย 0 (ผ่าน)
    — ข้อความบล็อก: ระบุชื่อ branch ปัจจุบัน + แนะนำการสร้าง feature/[slug] หรือ fix/[slug]
  3. Update templates/hooks/hooks.json: add PreToolUse array with "matcher": "Edit|Write" pointing to pre-edit-branch-guard.jsอัปเดต templates/hooks/hooks.json: เพิ่ม PreToolUse array ด้วย "matcher": "Edit|Write" ที่ชี้ไปยัง pre-edit-branch-guard.js
  4. Update agents/shared/devstarter-agent-base.md Branch Guard section: add one-line note about hookอัปเดตส่วน Branch Guard ของ agents/shared/devstarter-agent-base.md: เพิ่มหมายเหตุหนึ่งบรรทัดเกี่ยวกับ hook
  5. Update sdlc/devstarter-change.md Rule 9: add one-line note about hookอัปเดต Rule 9 ของ sdlc/devstarter-change.md: เพิ่มหมายเหตุหนึ่งบรรทัดเกี่ยวกับ hook
  6. Copy all modified files to ~/.claude/คัดลอกไฟล์ที่แก้ไขทั้งหมดไปยัง ~/.claude/
  7. Test: manually trigger Edit on develop → confirm block; switch to feature branch → confirm pass-throughทดสอบ: เรียก Edit บน develop ด้วยตนเอง → ยืนยันการบล็อก สลับไปยัง feature branch → ยืนยันการผ่าน
  8. Commit and open PR to developCommit และเปิด PR ไปยัง develop

Testing Approachแนวทางการทดสอบ

Manual functional test: (1) confirm git branch --show-current returns develop, attempt an Edit tool call, verify Claude receives the block decision and creates a work branch instead; (2) checkout feature/test-branch, attempt an Edit tool call, verify it passes through normally; (3) run in a non-git directory, verify pass-through with no error.การทดสอบฟังก์ชันด้วยตนเอง: (1) ยืนยันว่า git branch --show-current ส่งคืน develop, ลองเรียก tool Edit, ตรวจสอบว่า Claude ได้รับคำตัดสินบล็อกและสร้าง work branch แทน (2) checkout feature/test-branch, ลองเรียก tool Edit, ตรวจสอบว่าผ่านได้ตามปกติ (3) รันในไดเรกทอรีที่ไม่ใช่ git, ตรวจสอบว่าผ่านได้โดยไม่มีข้อผิดพลาด
