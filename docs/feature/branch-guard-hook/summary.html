Root Problem / Requirement

Original Problem / Requirement

Branch Guard (Rule 9) existed as text rules in devstarter-change.md and agents/shared/devstarter-agent-base.md only. The rule instructed Claude to check the branch before editing, but no technical mechanism enforced it. The AI model could still write files on protected branches (develop, main, uat) during autopilot mode, resume flows, or edge cases where the rule was overlooked.

Root Cause / Business Gap

No Claude Code PreToolUse hook existed to intercept Edit/Write tool calls before they executed. Enforcement was 100% documentation-based with no technical safety net. This meant a single missed rule check could result in commits directly on a protected branch.

How This Change Resolves It

Added scripts/hooks/pre-edit-branch-guard.js — a PreToolUse hook that runs before every Edit or Write tool call. It checks git branch --show-current and outputs a JSON block decision ({"decision":"block","reason":"..."}) if Claude is on a protected branch. The block message names the current branch and instructs Claude to create the correct work branch. Non-git directories pass through without error.

Implementation Notes

Key Decisions Made

  • Used {"decision":"block","reason":"..."} stdout JSON format (Claude Code PreToolUse block protocol) rather than non-zero exit, so the error message is human-readable in Claude's context
  • Exact match on protected branch names (no glob/prefix) to avoid false blocks on branches like develop-v2
  • git rev-parse --git-dir called first to detect non-git dirs; any exception causes pass-through to prevent false blocks
  • Block message is educational: names the current branch and tells Claude exactly which branch type to create
  • Hook registered in templates/hooks/hooks.json (not directly in .claude/settings.json) so it deploys correctly via install.sh --hooks

Technical Approach

The hook follows the same Node.js stdin/stdout pass-through pattern as stop-check-console-log.js. It reads stdin (up to 64KB), runs two synchronous execFileSync git calls (rev-parse to detect git repo, then branch --show-current), checks the result against a Set of protected branch names, and either outputs a block decision JSON or passes stdin through unchanged. The hook always exits 0 — Claude Code handles the block via the JSON decision field, not exit codes.

Deviations from Plan

None — implemented as planned. Hook logic, protected branch list, pass-through on non-git, and educational block message all match the consultation and plan.html spec exactly.

Handoff

👁 For Code Reviewers

What to Focus On

  • Hook exit path: ensure every code path exits 0 (never non-zero — Claude Code interprets non-zero as fatal tool failure, not a block)
  • Protected branch Set — verify all 5 names: develop, main, master, uat, dev
  • Non-git pass-through: the outer try/catch in getCurrentBranch() returns null — verify null check correctly skips the block
  • stdin MAX_STDIN cap (64KB) — appropriate for PreToolUse payload

Files Requiring Closest Review

  • scripts/hooks/pre-edit-branch-guard.js — core logic
  • templates/hooks/hooks.json — verify PreToolUse placement is top-level array item, not nested inside existing hooks

Known Trade-offs / Tech Debt

Hook adds ~30–50ms to every Edit/Write call (two git subprocesses). Acceptable for developer tooling. No automated unit tests — manual verification only. Future: add integration test in tests/hooks/.

✅ For QA / Testers

Test Scenarios

  1. On develop → attempt Edit tool call → block message appears, Claude creates work branch
  2. On main → same test → block message appears
  3. On uat → same test → block message appears
  4. On feature/any-name → Edit proceeds without block
  5. On fix/any-name → Edit proceeds without block
  6. Outside git repo → Edit proceeds without block or error

Edge Cases to Check

  • Branch named develop-experimental (should NOT be blocked — exact match only)
  • Detached HEAD state (branch name is empty — should pass through)
  • Very large PreToolUse payload (stdin cap at 64KB)

Regression Areas

  • All existing hooks still fire after PreToolUse hook is installed
  • install.sh --hooks still installs all 5 hooks correctly (no duplicates)

Environment Notes

Requires install.sh --hooks or manual node scripts/install-hooks.js to activate. Hook NOT active just from cloning the repo — must be installed into ~/.claude/settings.json.

📈 For Management

Business Impact

DevStarter now technically enforces branch safety for all AI-assisted development. The risk of accidental commits directly to protected branches (develop, main, uat) is eliminated — not just documented.

User Impact

Developers using DevStarter will see a clear error message if Claude tries to edit on a protected branch, with an instruction to create the right work branch. No change to normal workflow when already on a feature/fix branch.

Risk Assessment

Low. ~50ms overhead per file edit. Rollback: remove PreToolUse entry from hooks.json, re-run install.

What Was Delivered

  • Educational error message tells Claude exactly what to do next
  • Zero false blocks on feature/fix branches or non-git directories
