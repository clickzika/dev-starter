Requirementsความต้องการ

Problem / Gapปัญหา / ช่องว่าง

DevStarter v5.9.0 ships a knowledge vault with 4 note templates, an emit procedure (E1–E5), and SDLC integration across 7 workflows. Three gaps remain: (1) the vault is flat — no folders, no MOC index pages, no navigation beyond text search; (2) there is no skill to analyse an existing .md file and emit it — only manual one-at-a-time capture via /devstarter-knowledge; (3) wikilinks must be hand-written — no auto-discovery of related notes. DevStarter v5.9.0 มี knowledge vault พร้อม template 4 แบบ ขั้นตอน emit (E1–E5) และ integration กับ SDLC workflow 7 รายการ ยังมีช่องว่าง 3 ด้าน: (1) vault เป็นแบบ flat ไม่มีโฟลเดอร์ ไม่มีหน้า MOC index ไม่มีระบบนำทางนอกจากการค้นหาข้อความ (2) ไม่มีคำสั่งสำหรับวิเคราะห์ไฟล์ .md ที่มีอยู่และ emit มัน มีแค่การจับภาพด้วยตนเองครั้งละหนึ่งรายการผ่าน /devstarter-knowledge (3) wikilink ต้องเขียนด้วยมือ ไม่มีการค้นหาโน้ตที่เกี่ยวข้องโดยอัตโนมัติ

User StoryUser Story

As a DevStarter user, I want to drop any existing .md file into /devstarter-vault-ingest so that it is automatically classified, linked to related notes, and stored in the vault in the correct folder — without writing frontmatter manually. ในฐานะผู้ใช้ DevStarter ฉันต้องการนำไฟล์ .md ใดก็ได้เข้าสู่ /devstarter-vault-ingest เพื่อให้มันจัดประเภท เชื่อมโยงกับโน้ตที่เกี่ยวข้อง และบันทึกลงใน vault ในโฟลเดอร์ที่ถูกต้องโดยอัตโนมัติ

Acceptance Criteriaเกณฑ์การยอมรับ

  • Given folder_structure: hierarchical in config + E1 emit, When note emitted, Then lands in correct type folderเมื่อตั้งค่า hierarchical และ E1 ทำงาน โน้ตไปอยู่ในโฟลเดอร์ที่ถูกต้อง
  • Given vault scaffold, When complete, Then _index/HOME.md + 4 MOC files exist with Dataview blocksเมื่อ scaffold เสร็จ HOME.md + 4 MOC มีอยู่พร้อม Dataview blocks
  • Given /devstarter-vault-ingest notes/finding.md, When run, Then frontmatter proposed at confirmation gate before emitเมื่อรัน vault-ingest frontmatter ถูกเสนอที่ gate ก่อน emit
  • Given existing vault notes, When ingest runs, Then related notes surfaced + wikilinks proposedเมื่อมีโน้ตใน vault อยู่ ingest จะแสดงโน้ตที่เกี่ยวข้องและเสนอ wikilink
  • Given file with secrets, When ingest runs, Then E4 deny-list blocks the secret before writeเมื่อไฟล์มีข้อมูลลับ E4 deny-list บล็อกก่อนเขียน
  • All 4 existing Obsidian templates have ## Links sectiontemplate ทั้ง 4 มีส่วน Links
  • /devstarter-vault-ingest registered as item 46 in devstarter-menu.mdลงทะเบียนเป็นรายการ 46 ใน menu
  • Wikilink format [[folder/slug]] documented in docs/obsidian-vault-guide.mdรูปแบบ wikilink ถูกบันทึกใน obsidian-vault-guide.md

Solution Designการออกแบบ

Part A — Vault Structureส่วน A — โครงสร้าง Vault

vault/
├── bugs/           ← type:bug notes
├── techniques/     ← type:technique notes
├── rcas/           ← type:rca notes
├── snapshots/      ← type:project-snapshot notes
└── _index/
    ├── HOME.md     ← Obsidian startup note, links to all MOCs
    ├── MOC-bugs.md
    ├── MOC-techniques.md
    ├── MOC-rcas.md
    └── MOC-projects.md

Each MOC page uses a Dataview query block to auto-list notes in its folder. The emit procedure E1 gains a new branch: if folder_structure: hierarchical, it resolves the destination path as <vault_path>/<subdir>/<type>s/<slug>.md rather than <vault_path>/<subdir>/<slug>.md. Flat mode (current default) is unchanged. A new folder_structure: flat value preserves existing behaviour explicitly. แต่ละหน้า MOC ใช้ Dataview query block เพื่อแสดงโน้ตในโฟลเดอร์ของมันโดยอัตโนมัติ ขั้นตอน E1 ได้รับ branch ใหม่: ถ้าตั้งค่า folder_structure: hierarchical จะกำหนดเส้นทางปลายทางเป็น <vault_path>/<subdir>/<type>s/<slug>.md แทนที่จะเป็น <vault_path>/<subdir>/<slug>.md โหมด flat (ค่าเริ่มต้นปัจจุบัน) ไม่เปลี่ยนแปลง

Part B — /devstarter-vault-ingest Skill (4-phase flow)ส่วน B — คำสั่ง /devstarter-vault-ingest (4 phases)

PhaseActionการกระทำOutputผลลัพธ์
1 — AnalyzeRead MD file → infer type (bug/rca/technique/snapshot) from content shape → extract title, language, framework, tagsอ่านไฟล์ MD → อนุมานประเภทจากรูปแบบเนื้อหา → ดึง title, language, framework, tagsProposed frontmatter shown at AskUserQuestion gatefrontmatter ที่เสนอแสดงที่ gate
2 — DiscoverGrep vault by tags + language + root_cause_category → surface 1–5 matching notesGrep vault ตาม tags + language + category → แสดงโน้ตที่ตรงกัน 1–5 รายการProposed [[wikilinks]] in ## Links sectionwikilink ที่เสนอในส่วน Links
3 — EmitSanitize via E4 deny-list → write to vault/<type>s/<slug>.mdSanitize ผ่าน E4 deny-list → เขียนไปยัง vaultNote file created in vaultสร้างไฟล์โน้ตใน vault
4 — ConfirmShow emitted path + linked notesแสดงเส้นทางที่บันทึกและโน้ตที่เชื่อมโยงUser confirmation messageข้อความยืนยันผู้ใช้

Wikilink ConventionConvention ของ Wikilink

  • Format: [[<folder>/<slug>]] — no spaces, kebab-case slug (e.g. [[bugs/auth-token-expiry]])รูปแบบ: [[<folder>/<slug>]] ไม่มีเว้นวรรค ใช้ kebab-case
  • Cross-type links allowed: a bug note can link to a technique that fixed itลิงก์ข้ามประเภทได้: โน้ต bug สามารถลิงก์ไปยัง technique ที่แก้ไขมัน
  • Every template has a ## Links section (empty by default, filled by ingest)ทุก template มีส่วน ## Links (ว่างเปล่าโดยค่าเริ่มต้น เติมโดย ingest)

Testingการทดสอบ

Verification Stepsขั้นตอนการตรวจสอบ

  1. Set folder_structure: hierarchical in devstarter-config.yml. Run /devstarter-knowledge to emit a bug note. Verify it lands in <vault_path>/knowledge/bugs/<slug>.md.ตั้งค่า hierarchical และรัน /devstarter-knowledge เพื่อ emit bug note ตรวจสอบว่าอยู่ในโฟลเดอร์ bugs/
  2. Run /devstarter-vault-ingest <any-existing-md>. Verify Phase 1 proposes frontmatter at a gate before writing anything.รัน /devstarter-vault-ingest บนไฟล์ MD ที่มีอยู่ ตรวจสอบว่า Phase 1 เสนอ frontmatter ที่ gate ก่อนเขียน
  3. With multiple notes in vault, run ingest on a bug note. Verify Phase 2 surfaces related notes and proposes wikilinks in the ## Links section.เมื่อมีโน้ตหลายรายการใน vault รัน ingest บน bug note ตรวจสอบว่า Phase 2 แสดงโน้ตที่เกี่ยวข้องและ wikilink
  4. Open _index/HOME.md in Obsidian. Verify the 4 MOC links resolve and Dataview queries return existing notes.เปิด HOME.md ใน Obsidian ตรวจสอบว่า 4 MOC ลิงก์ทำงานและ Dataview queries คืนโน้ตที่มีอยู่
  5. Revert config to folder_structure: flat. Run /devstarter-knowledge again. Verify note still emits to flat path (no regression).Revert กลับเป็น flat และรัน /devstarter-knowledge อีกครั้ง ตรวจสอบว่าโน้ต emit ไปยัง flat path (ไม่มี regression)
