Overviewภาพรวม

📝 What you are being asked to approve

We are adding a clean folder-based structure to the Obsidian knowledge vault and creating a new /devstarter-vault-ingest skill that reads any existing Markdown file, classifies it automatically, finds related vault notes, and stores it with auto-generated wikilinks — making knowledge recall fast and reliable. เราจะเพิ่มโครงสร้างโฟลเดอร์ที่ชัดเจนให้กับ Obsidian knowledge vault และสร้างคำสั่งใหม่ /devstarter-vault-ingest ที่อ่านไฟล์ Markdown ใดก็ได้ จัดประเภทโดยอัตโนมัติ ค้นหาโน้ตที่เกี่ยวข้องในวอลต์ และบันทึกพร้อม wikilink ที่สร้างขึ้นอัตโนมัติ เพื่อให้การค้นหาความรู้รวดเร็วและเชื่อถือได้

What We Will Buildสิ่งที่เราจะสร้าง

👤 For the requester — please confirm this matches your intentสำหรับผู้ร้องขอ — โปรดยืนยันว่าตรงกับความต้องการ

The vault currently stores all notes in a flat structure with no folders or navigation. Engineers write notes but cannot find them once the count grows past ~20. We will add a folder hierarchy (bugs/, techniques/, rcas/, snapshots/) with MOC (Map of Content) index pages that give an always-fresh view of every note type via Dataview queries. We will also create a new /devstarter-vault-ingest <file> command that reads any existing Markdown file, infers its type and metadata, finds related notes in the vault, and emits it with auto-generated wikilinks — making the vault a genuine recall tool, not just an archive. ปัจจุบัน vault เก็บโน้ตทั้งหมดในโครงสร้างแบน ไม่มีโฟลเดอร์หรือระบบนำทาง วิศวกรเขียนโน้ตแต่ค้นหาไม่เจอเมื่อจำนวนมากกว่า 20 รายการ เราจะเพิ่มโครงสร้างโฟลเดอร์ (bugs/, techniques/, rcas/, snapshots/) พร้อมหน้า MOC (Map of Content) ที่แสดงโน้ตทุกประเภทแบบ real-time ผ่าน Dataview queries นอกจากนี้เรายังสร้างคำสั่งใหม่ /devstarter-vault-ingest <file> ที่อ่านไฟล์ Markdown ใดก็ได้ อนุมานประเภทและ metadata ค้นหาโน้ตที่เกี่ยวข้องใน vault และบันทึกพร้อม wikilink ที่สร้างอัตโนมัติ

User StoryUser Story

As a DevStarter user, I want to drop any existing .md file into /devstarter-vault-ingest so that it is automatically classified, linked to related notes, and stored in the vault in the correct folder — without writing frontmatter manually. ในฐานะผู้ใช้ DevStarter ฉันต้องการนำไฟล์ .md ใดก็ได้เข้าสู่ /devstarter-vault-ingest เพื่อให้มันจัดประเภท เชื่อมโยงกับโน้ตที่เกี่ยวข้อง และบันทึกลงใน vault ในโฟลเดอร์ที่ถูกต้องโดยอัตโนมัติ โดยไม่ต้องเขียน frontmatter ด้วยตนเอง

  • Vault scaffold: bugs/ techniques/ rcas/ snapshots/ _index/โครงสร้าง vault: bugs/ techniques/ rcas/ snapshots/ _index/
  • MOC pages: HOME.md + 4 type MOCs with Dataview queriesหน้า MOC: HOME.md + 4 ประเภท พร้อม Dataview queries
  • 4 existing templates updated with ## Links sectionอัปเดต 4 template เดิมด้วยส่วน ## Links

Acceptance Criteria — done whenเกณฑ์การยอมรับ — เสร็จเมื่อ

  • Given folder_structure: hierarchical in config, When E1 emit runs, Then note lands in vault/bugs/, vault/techniques/, vault/rcas/, or vault/snapshots/ based on typeเมื่อตั้งค่า folder_structure: hierarchical และ E1 ทำงาน โน้ตจะไปอยู่ในโฟลเดอร์ที่ถูกต้องตามประเภท
  • Given vault scaffold runs, When complete, Then _index/HOME.md and 4 MOC files exist with working Dataview blocksเมื่อ vault scaffold ทำงานเสร็จ _index/HOME.md และ MOC 4 ไฟล์มีอยู่พร้อม Dataview blocks ที่ใช้งานได้
  • Given /devstarter-vault-ingest notes/my-finding.md, When run, Then Claude proposes classified frontmatter at a confirmation gate before emittingเมื่อรัน /devstarter-vault-ingest notes/my-finding.md Claude จะเสนอ frontmatter ที่จัดประเภทแล้วก่อนที่จะบันทึก
  • Given vault has existing notes, When ingest runs, Then related notes are surfaced and wikilinks proposed before emitเมื่อ vault มีโน้ตอยู่และ ingest ทำงาน โน้ตที่เกี่ยวข้องจะถูกแสดงและ wikilink จะถูกเสนอก่อนบันทึก
  • Given a file with secrets, When ingest runs, Then sanitize E4 deny-list blocks the secret before writeเมื่อไฟล์มีข้อมูลลับและ ingest ทำงาน deny-list E4 จะบล็อกข้อมูลลับก่อนเขียน
  • All 4 existing Obsidian templates have a ## Links sectiontemplate Obsidian ทั้ง 4 มีส่วน ## Links
  • /devstarter-vault-ingest appears as item 46 in devstarter-menu.md/devstarter-vault-ingest ปรากฏเป็นรายการ 46 ใน devstarter-menu.md
  • Wikilink format [[<folder>/<slug>]] documented in docs/obsidian-vault-guide.mdรูปแบบ wikilink [[<folder>/<slug>]] ถูกบันทึกใน docs/obsidian-vault-guide.md

Management Summaryสรุปสำหรับผู้บริหาร

📊 For management — business rationale, no technical detailสำหรับผู้บริหาร — เหตุผลเชิงธุรกิจ ไม่มีรายละเอียดทางเทคนิค

Why now — the business needทำไมต้องทำตอนนี้ — ความจำเป็นทางธุรกิจ

DevStarter's knowledge vault was built to capture engineering know-how across projects. But without folders or navigation, it becomes a write-only graveyard after 20 notes — engineers write but cannot recall. This change adds organisation and a one-command capture tool so the vault delivers on its core promise: reusable knowledge. knowledge vault ของ DevStarter ถูกสร้างเพื่อรวบรวมความรู้ทางวิศวกรรมข้ามโปรเจกต์ แต่ถ้าไม่มีโฟลเดอร์หรือระบบนำทาง มันจะกลายเป็นที่เก็บที่ค้นหาไม่ได้หลังจากโน้ต 20 รายการ การเปลี่ยนแปลงนี้เพิ่มการจัดระเบียบและเครื่องมือบันทึกแบบคำสั่งเดียว เพื่อให้ vault ตอบสนองต่อจุดประสงค์หลัก: ความรู้ที่นำกลับมาใช้ได้

Who benefitsใครได้ประโยชน์

Every developer using DevStarter across multiple projects. The vault becomes the team's shared brain — past bugs, proven techniques, and project snapshots are instantly searchable rather than buried in flat files. นักพัฒนาทุกคนที่ใช้ DevStarter ข้ามโปรเจกต์ vault จะกลายเป็นสมองร่วมของทีม — bugs ในอดีต เทคนิคที่พิสูจน์แล้ว และ snapshot ของโปรเจกต์สามารถค้นหาได้ทันทีแทนที่จะถูกฝังในไฟล์แบน

Cost of not doing thisผลกระทบหากไม่ดำเนินการ

Without this change, the vault feature degrades with use. Engineers will stop writing notes when recall fails. Cross-project knowledge reuse — the primary value proposition — will not be realised, and the vault becomes technical debt rather than an asset. หากไม่ดำเนินการ ฟีเจอร์ vault จะเสื่อมลงตามการใช้งาน วิศวกรจะหยุดเขียนโน้ตเมื่อการค้นคืนล้มเหลว การนำความรู้ข้ามโปรเจกต์กลับมาใช้ — ซึ่งเป็นคุณค่าหลัก — จะไม่เกิดขึ้น และ vault จะกลายเป็นหนี้เทคนิคแทนที่จะเป็นทรัพย์สิน
