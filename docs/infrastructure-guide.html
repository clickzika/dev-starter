Architecture Overview

DevStarter has no cloud infrastructure. It is a local tool that runs inside Claude Code sessions. Its "infrastructure" is the filesystem at ~/.claude/ and the git remote topology used to publish releases.

Installation

One-Command Install

git clone https://github.com/clickzika/dev-starter.git ~/.claude/DevStarter
bash ~/.claude/DevStarter/install.sh

What install.sh Does

  1. Copies all files from the repo to ~/.claude/
  2. Merges permissions on existing ~/.claude/settings.json (does not overwrite)
  3. Creates ~/.claude/USER.md via a setup wizard (name, role, skill level)
  4. Creates .env.example at the project root
  5. Adds .env to .gitignore if not already present

Release Pipeline

DevStarter uses Strategy I (git auto-detect) from /devstarter-release. The dual-remote setup is detected at runtime.

flowchart LR A[develop branch] --> B[PR to main] B --> C[Merge to main] C --> D[Bump VERSION file] D --> E[Update CHANGELOG.md] E --> F[git commit] F --> G[git tag vX.Y.Z] G --> H[git push origin main] G --> I[git push release main] G --> J[git push release vX.Y.Z]

publish.sh Steps

#!/bin/bash
set -euo pipefail
VERSION=$(cat VERSION)
git tag "v$VERSION"
git push origin main
git push release main
git push release "v$VERSION"
echo "✅ v$VERSION published to release remote"

Git Remote Topology

RemoteURLPurposeWho pushes
origingithub.com/clickzika/dev-starter-devDevelopment repo — PRs, issues, CIDeveloper (daily)
releasegithub.com/clickzika/dev-starterPublic distribution — users clone thispublish.sh (on release only)
Rule

Never push directly to the release remote from a feature branch. Only main is pushed to release, and only via publish.sh after explicit version tagging.

Currently no CI pipeline runs on the dev repo — validation is manual. Future work (OI-001): add a linting/validation workflow for runbook correctness.

Update Runbook

Run /devstarter-update to update an existing installation:

  1. Agent reads current VERSION from disk
  2. Fetches latest from release remote: git fetch release
  3. Shows diff summary: new commands, changed runbooks, new agents
  4. User approves update
  5. Agent runs install.sh again (safe — merges, does not overwrite USER.md or .project.env)
  6. Agent updates VERSION and announces what changed
7

Rollback Runbook

If an update breaks a workflow:

  1. Identify the last known good version: git tag --sort=-version:refname | head -5
  2. Checkout that version from release remote: git checkout release/v1.4.0 -- sdlc/ agents/ commands/
  3. Re-run install.sh to restore files to ~/.claude/
  4. Update VERSION file to match restored version
  5. Document the rollback in CHANGELOG.md
