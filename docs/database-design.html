Overview

DevStarter has no relational database. Its persistent state is stored in a small set of plain-text configuration and JSON files. This document defines the schema, purpose, read/write rules, and constraints for each data file used by the DevStarter workflow system.

Note

DevStarter is a toolkit installed at ~/.claude/, not an application with a database. "Schema" here means the structure of its configuration and state files.

FileFormatPurposeScope
memory/progress.jsonJSONWorkflow checkpoint — current gate, last/next task, files modifiedPer project
.project.envKey=ValueProject configuration — VCS, PM, CI, secrets, AI providerPer project

progress.json Schema

Written to memory/progress.json in the project root after every workflow task. Used by the Cron auto-resume job and manual resume to continue interrupted work.

{
  "workflow":         string,    // Required. Workflow name e.g. "devstarter-change"
  "status":           string,    // Required. "in_progress" | "completed" | "idle" | "interrupted"
  "gate":             number,    // Optional. Current gate number (1–5)
  "current_task":     string,    // Required. Human-readable current task name
  "completed_tasks":  string[],  // Required. Ordered list of completed task names
  "next_task":        string,    // Required. What to do on resume
  "files_modified":   string[],  // Required. Files written this session
  "last_checkpoint":  string,    // Required. ISO 8601 timestamp e.g. "2026-04-05T14:30:00"
  "notes":            string,    // Optional. Human-readable context for resuming agent

  // Optional fields for release/version workflows:
  "version":          string,    // SemVer e.g. "1.4.1"
  "branch":           string,    // Current git branch
  "feature_in_progress": string | null,
  "recently_shipped": string[],  // Last N shipped features
  "completed_at":     string,    // ISO 8601, set when status = "completed"
  "session":          string,    // Date string e.g. "2026-04-05"
  "next_file_to_create": string | null,
  "next_action":      string     // Human-readable "what to do next" for idle state
}

Status Transitions

stateDiagram-v2 [*] --> idle : workflow completed or first install idle --> in_progress : workflow started in_progress --> in_progress : task completed (checkpoint saved) in_progress --> interrupted : rate limit hit / session closed interrupted --> in_progress : Cron auto-resume or user resumes in_progress --> completed : all gates passed completed --> idle : cleanup

.project.env Schema

Committed to the project repo (no secrets). Defines integration targets for VCS, project management, CI/CD, and AI provider. Read by DevOps and PM agents to route to the correct procedures.

# VERSION CONTROL
VCS_TYPE=github                          # github | gitlab | svn | azure-devops | bitbucket
GITHUB_REPO=owner/repo-name              # e.g. clickzika/dev-starter-dev
VCS_SECONDARY_1=none                     # mirror target 1 (same options as VCS_TYPE)
VCS_SECONDARY_2=none                     # mirror target 2
VCS_SYNC_BRANCHES=main develop           # space-separated branch list to mirror

# BRANCHING STRATEGY
BRANCH_STRATEGY=gitflow                  # gitflow | trunk | github-flow
MAIN_BRANCH=main
DEV_BRANCH=develop

# PROJECT MANAGEMENT
PM_TYPE=notion                           # notion | jira | linear | github-projects | none
NOTION_DATABASE_ID=xxx-xxx-xxx           # Notion DB UUID
NOTION_BOARD_URL=https://notion.so/...

# Jira fields (if PM_TYPE=jira)
JIRA_PROJECT_KEY=
JIRA_BOARD_ID=
JIRA_SPRINT_ID=
JIRA_DEFAULT_ISSUE_TYPE=Story
JIRA_STORY_POINTS_FIELD=story_points
JIRA_EMAIL=

# CI/CD
CI_TYPE=github-actions                   # github-actions | gitlab-ci | jenkins | none

# SECRETS BACKEND
SECRETS_BACKEND=env-file                 # env-file | aws-secrets-manager | azure-keyvault | gcp-secret-manager | vault

# AI PROVIDER
AI_PROVIDER=claude                       # claude | openai | gemini | azure-openai | litellm

# PROJECT TYPE
PROJECT_TYPE=framework                   # framework | web | api | mobile | ml | cli | desktop
LANGUAGE=markdown+bash

CLAUDE.md Schema

A Markdown file placed at the project root. Claude Code loads it automatically at every session. All agents must read it before doing any work. It is the single source of truth for project conventions.

Required Sections

SectionRequiredContent
## ProjectYesName, description, repo URL, Notion board, version, status
## Session StartYesOrdered file reading list for every new session
## Session Resume ProtocolYesHow to detect and resume interrupted workflows
## Tech Stack / Project TypeYesLanguage, framework, database, auth method
## Project StructureRecommendedDirectory tree with purpose annotations
## VCS ConfigurationYesRepo URL, branch strategy, naming conventions
## Workflow RulesYesGate approval rules, commit conventions, naming rules
## PM ConfigurationRecommendedNotion/Jira board URL, task status flow

Data Rules & Constraints

Rule IDRuleEnforcement
DR-001progress.json MUST be written after every task before any other operationAll SDLC runbooks
DR-002.env MUST be listed in .gitignore — never committedinstall.sh, .gitignore template
DR-003CLAUDE.md is the single source of truth — agents read from disk, never chat historyAll agent specs
DR-004.project.env MUST be committed — it contains no secretsConvention
DR-005progress.json status MUST be "completed" when a workflow finishes, not left as "in_progress"Cleanup step in every runbook
DR-006All timestamps in progress.json MUST be ISO 8601 formatConvention
DR-007VERSION file MUST contain only the bare SemVer string (e.g. 1.4.1)publish.sh
DR-008A workflow checkpoint MUST include the next_task field so resume can proceed without human input until the next gateAll SDLC runbooks
