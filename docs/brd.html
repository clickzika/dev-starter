Executive Summary

Problem Statement

Developers using Claude Code lack a consistent, repeatable engineering workflow. Every new project starts from zero: no agreed agent roles, no gate approvals, no SDLC runbooks, no document standards. This leads to inconsistent outputs, lost context between sessions, and no institutional process to build on.

Solution

DevStarter is a drop-in workflow system for Claude Code. Installing it gives any project a complete software team — 13 specialist AI agents, 24 slash commands, and 28+ SDLC runbooks — with a 5-gate approval workflow, session-resumable checkpoints, and a self-contained document portal.

Business Objectives

  • OBJ-001: Reduce time-to-first-working-feature from hours to minutes for any new project
  • OBJ-002: Eliminate context loss between Claude Code sessions via structured checkpoints
  • OBJ-003: Standardise document output so every project produces the same 9 core documents
  • OBJ-004: Support any technology stack without requiring per-stack customisation
  • OBJ-005: Be installable in under 60 seconds on Windows, macOS, and Linux

Scope

In Scope

  • 13 specialist AI agent definition files (Markdown)
  • 24 slash commands covering the full SDLC
  • 28+ SDLC workflow runbooks
  • HTML document templates (portal + 9 doc types)
  • GitHub and GitLab procedure runbooks (PROC-GH-01–17, PROC-GL-01–17)
  • Notion and Jira task-management integration procedures
  • SVN and multi-VCS sync support
  • MLOps agent and AI/ML project workflows
  • Enterprise secrets management templates (Vault, AWS, Azure, GCP)
  • LiteLLM multi-provider AI routing config
  • One-command installer (bash install.sh)
  • Checkpoint/auto-resume protocol for rate-limit recovery

Out of Scope

  • Hosting or cloud deployment of DevStarter itself (it is a local tool)
  • A web UI for DevStarter configuration
  • Native IDE plugin (VS Code extension, JetBrains plugin)
  • Automated test execution (DevStarter authors runbooks; projects execute their own tests)
  • Billing or licence management

Assumptions

  • Users have Claude Code installed and authenticated
  • Users have Git installed and configured
  • Users have bash available (Git Bash on Windows, native on macOS/Linux)
  • GitHub or GitLab access is available for VCS operations

Constraints

  • All file content must be plain Markdown or Bash — no compiled binaries
  • All file names must carry the devstarter- prefix (except root files)
  • Installation target is ~/.claude/ — no system-level changes
  • Must not break existing ~/.claude/ configurations (permissions merge, not overwrite)

AS-IS State

Current Process (without DevStarter)

A developer opening Claude Code for a new project faces an unstructured session. They must manually:

  • Define what they want built (no structured intake)
  • Ask Claude to perform specific tasks (no agent specialisation)
  • Remember what was done in prior sessions (no checkpoint system)
  • Create and format documents ad hoc (no template standards)
  • Manage GitHub issues, branches, and PRs manually (no procedure runbooks)
  • Re-explain the project on every new session (no persistent CLAUDE.md)

Pain Points

Pain PointImpactRoot Cause
Context loss between sessionsHigh — rework, repetitionNo persistent project state file
No structured workflowHigh — inconsistent outputsClaude is general-purpose; no SDLC layer
No agent specialisationMedium — shallow expertiseSingle-agent responses lack domain depth
Rate limit interruptionsMedium — lost progressNo checkpoint/resume mechanism
No document standardsMedium — every project looks differentNo shared template or portal
Manual GitHub/Notion opsLow — time-consumingNo procedure runbooks

TO-BE State

Desired Future State

With DevStarter installed, a developer opens Claude Code and immediately has access to a complete, structured workflow. They type a slash command, select an option, and a specialist agent takes over with deep domain knowledge, a gated approval workflow, and automatic session-resumable progress tracking.

flowchart LR A[Developer opens Claude Code] --> B[/devstarter-menu] B --> C{Pick workflow} C --> D[/devstarter-new] C --> E[/devstarter-change] C --> F[/devstarter-existing] D --> G[Specialist agents] E --> G F --> G G --> H[Gate approval] H --> I[Checkpoint saved] I --> J[Continue or resume]

Business Rules

  • BR-001: Every workflow MUST stop at each gate and wait for explicit user approval before proceeding
  • BR-002: All agent output MUST be saved to disk before the checkpoint is updated
  • BR-003: All document files MUST use the standard HTML template — never plain Markdown
  • BR-004: All new files MUST carry the devstarter- prefix
  • BR-005: A new slash command requires both a commands/ file and an sdlc/ runbook
  • BR-006: Version bumps follow Semantic Versioning: MAJOR.MINOR.PATCH

User ConfigUSER.mdUser skill level, preferences, communication styleinstall.sh wizard
Secrets.envLocal API keys and secrets (gitignored)User / secrets command

progress.json Schema

{
  "workflow":         string,   // e.g. "devstarter-change"
  "status":           string,   // "in_progress" | "completed" | "idle"
  "gate":             number,   // current gate number
  "current_task":     string,   // e.g. "BA — docs/brd.html"
  "completed_tasks":  string[], // list of completed task names
  "next_task":        string,   // what to do on resume
  "files_modified":   string[], // files written this session
  "last_checkpoint":  string,   // ISO 8601 timestamp
  "notes":            string    // human-readable context
}

Data Rules

  • DR-001: progress.json MUST be written after every task — before any other operation
  • DR-002: .env MUST be listed in .gitignore — never committed
  • DR-003: CLAUDE.md is the single source of truth for project stack and conventions
  • DR-004: .project.env MUST be committed to the repo — it contains no secrets

Glossary

TermDefinition
AgentA specialist AI persona defined in a Markdown file under agents/, read by Claude Code to shape its behaviour for a domain (e.g. Backend, QA, Security)
CheckpointA snapshot of current workflow progress written to memory/progress.json after each completed task, enabling resume after interruption
CLAUDE.mdA project-level context file placed at the repo root; Claude Code loads it automatically at every session start
GateA mandatory approval stop in a workflow where Claude halts and waits for the user to type "approve" before proceeding
GitFlowA branching strategy with main, develop, feature/*, hotfix/*, and release/* branches
RunbookA step-by-step procedure file in sdlc/ that defines a complete workflow (e.g. how to create a GitHub PR, how to run a sprint retrospective)
Slash commandA Claude Code command file in commands/ that users invoke with /devstarter-X; it routes to the matching SDLC runbook
SDLCSoftware Development Lifecycle — the end-to-end process from requirements to deployment
devstarter- prefixThe mandatory namespace prefix applied to all agents, commands, and SDLC files in this project to distinguish them from other Claude Code extensions
