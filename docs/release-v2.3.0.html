Overview
What this release adds and why
v2.3.0 establishes a complete 3-branch git strategy for every project DevStarter creates or onboards. New projects now automatically get main, uat, and develop branches — with develop set as the GitHub default. Branch protection rules (no force push, no deletion, PR review required) are applied to main and uat at setup time, and optionally to develop after scaffold. Existing repos get a new one-command PROC-GH-18 to catch up. A team-ready git-workflow.md reference document rounds out the release.
🌟 Key Benefits
  • 🌿
    3-branch strategy out of the box Every new project gets main → uat → develop wired automatically, with develop as the GitHub default branch.
  • 🔒
    Standard protection rules applied at Gate 0 main and uat are protected immediately: no force push, no deletion, 1 approving review required. No manual steps.
  • ⚙
    Optional develop protection for teams Post-scaffold, teams of 3+ are prompted whether to protect develop too — enforcing PR-based workflow for all devs.
  • 🔄
    Existing repos can catch up with PROC-GH-18 Run /devstarter-existing on any existing project to apply standard protection idempotently without re-creating anything.
  • 📄
    Team handoff document included docs/git-workflow.md covers the full flow — daily dev, PR review, release, hotfix, and a Quick Reference block.

What's New
New procedures, updated gates, and new team reference doc
🌿 PROC-GH-01 — 3-Branch Bootstrap MODIFIED
After creating the GitHub repo, PROC-GH-01 now pushes three branches and sets develop as the default branch via gh repo edit --default-branch develop. Gate 0 confirmation output shows all three branches and the default star:
✓ Branches: main (production) → uat (acceptance) → develop (default ★) ✓ Default branch: develop
🔒 PROC-GH-10 Step 1 — Standard Protection for main + uat MODIFIED
Protection payload now includes three additional fields applied to both main and uat:
RuleValueEffect
allow_force_pushesfalseBlocks git push --force
allow_deletionsfalsePrevents branch deletion
required_status_checksstrict: trueBranch must be up to date before merge
required_pull_request_reviewscount: 1, dismiss_stale: true1 approving review required; stale reviews dismissed on new commits
⚙ PROC-GH-10 Step 2 — Optional develop Protection NEW
Prompted at Gate 3 completion (after scaffold). Applies the same protection payload to develop. Recommended for teams of 3 or more — enforces the feature/* → PR → develop workflow for all developers. Solo developers or small teams can skip this and push directly to develop.
✓ develop branch: protected — all devs use PRs — or — ✓ develop branch: unprotected — direct push allowed
🔄 PROC-GH-18 — Apply Protection to Existing Repos NEW
New idempotent procedure for repos that were created before v2.3.0. Reads main_branch and uat_branch from devstarter-config.yml, checks that each branch exists before applying, then applies the standard Step 1 protection payload via GitHub API PUT. Wired into /devstarter-existing Phase 3.5 Step 3 automatically — runs after PROC-GH-02. Safe to run multiple times (idempotent).
📄 docs/git-workflow.md — Team Handoff Reference NEW
A standalone Markdown file in docs/ that every team member can read. Covers:
  • Branch Overview table (main / uat / develop — purpose, who pushes, protection status)
  • Daily Dev Workflow — branch from develop, commit, push, open PR
  • Head Dev PR review and merge commands (gh pr list, gh pr review, gh pr merge)
  • Release Flow — develop → uat → main with exact git commands
  • Hotfix Flow — branch from main, fix, PR, tag, backport to uat + develop
  • Branch Protection Rules table
  • 7 Key Rules and Quick Reference block
🔧 devstarter-config.yml — uat_branch field MODIFIED
Added uat_branch: uat field alongside existing main_branch and dev_branch. sync_branches updated to "main uat develop". PROC-GH-10 Step 2 and PROC-GH-18 both read from this config as the single source of truth for branch names.

How to Update
From v2.2.x to v2.3.0
Run /devstarter-update
In any Claude Code session, run:
/devstarter-update
This pulls the latest version and installs all new files to ~/.claude/. The agents/custom/ folder is preserved during updates.
Apply protection to existing projects
For projects already running with DevStarter, run PROC-GH-18 once to apply the new protection rules:
# In your existing project /devstarter-existing
Claude will run Phase 3.5 Step 3 which calls PROC-GH-18 automatically. Safe to run on already-protected repos.
No breaking changes
v2.3.0 is fully backward compatible. PROC-GH-01 changes apply only to new projects. PROC-GH-10 protection payload changes are additive. PROC-GH-18 is idempotent. All existing commands, agents, and config files continue to work without modification.

Usage Examples
3-branch workflow in practice
New project — all branches created automatically
# Start a new project — Gate 0 handles everything /devstarter-new # Gate 0 output will show: # ✓ GitHub: github.com/you/my-project # ✓ Branches: main → uat → develop (default ★) # ✓ Branch protection: main + uat protected # ✓ Default branch: develop
Protect develop for a team
After scaffold completes (Gate 3), you'll be asked:
# Prompted at Gate 3: # "Protect the develop branch? (recommended for teams ≥ 3)" # Type: yes → protection applied # Type: no → direct push to develop allowed # All devs then work via: git checkout -b feature/my-task git push origin feature/my-task gh pr create --base develop
Catch up an existing repo
# In an existing project that has devstarter-config.yml /devstarter-existing # Phase 3.5 Step 3 will run PROC-GH-18: # ✓ Branch protection: main + uat — PR required, no force push, no deletion
Share git-workflow.md with your team
# The file is in your project's docs/ folder # Share directly or link from README: cat docs/git-workflow.md # Or commit and share the GitHub link: # github.com/you/my-project/blob/develop/docs/git-workflow.md

Known Issues
Planned for v2.3.1
  • 🔶
    PROC-GH-18 not wired for GitLab / SVN PROC-GH-18 only applies to GitHub repos. GitLab protected branches and SVN lock policies are not yet automated. Manual setup required for non-GitHub VCS. Planned for v2.3.1.
  • 🔶
    required_status_checks contexts always empty The protection payload sets strict: true but contexts: [] — no specific CI checks are required. Once a project has CI configured, users must manually add check names to the protection rule. Automated CI check wiring planned for v2.4.0.
  • 🔶
    docs/git-workflow.md not auto-generated per project The file exists in the DevStarter repo itself but is not yet templated and copied into each generated project's docs/ folder at Gate 3. Planned for v2.3.1.
