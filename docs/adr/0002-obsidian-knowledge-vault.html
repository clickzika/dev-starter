Contextบริบท

A team uses DevStarter to build many projects and wants captured engineering knowledge — know-how, bugs, root causes, and how they were fixed — kept in one shared Obsidian vault, so a technique or root cause found on project A can be reused on project B. DevStarter is already half-built for this: /devstarter-postmortem and /devstarter-bug-postmortem already produce root-cause records, and DevStarter's own memory/ files already use YAML frontmatter + [[wikilink]] + one-fact-per-file — which is exactly the Obsidian note format. The gap is that those records are per-project HTML in docs/, not a shared, cross-project searchable Markdown knowledge base.ทีมหนึ่งใช้ DevStarter สร้างหลายโปรเจกต์ และต้องการเก็บความรู้เชิงวิศวกรรมที่จับได้ — know-how, บั๊ก, ต้นตอของปัญหา และวิธีแก้ — ไว้ใน Obsidian vault ที่แชร์กันที่เดียว เพื่อให้เทคนิคหรือต้นตอที่เจอในโปรเจกต์ A นำไปใช้ซ้ำในโปรเจกต์ B ได้ DevStarter มีพื้นฐานพร้อมอยู่แล้วครึ่งหนึ่ง: /devstarter-postmortem และ /devstarter-bug-postmortem สร้างบันทึกต้นตอปัญหาอยู่แล้ว และไฟล์ memory/ ของ DevStarter เองก็ใช้ YAML frontmatter + [[wikilink]] + หนึ่งข้อเท็จจริงต่อไฟล์ ซึ่งเป็นรูปแบบโน้ตของ Obsidian พอดี ช่องว่างคือบันทึกเหล่านั้นเป็น HTML ราย-โปรเจกต์ใน docs/ ไม่ใช่คลังความรู้ Markdown ที่แชร์และค้นข้ามโปรเจกต์ได้

📎 Constraintsข้อจำกัด
  • DevStarter is pure Markdown + Bash — no build step, no runtime. The emitter must be Markdown/Bash, not a service.DevStarter เป็น Markdown + Bash ล้วน — ไม่มีขั้นตอน build ไม่มี runtime ตัว emitter ต้องเป็น Markdown/Bash ไม่ใช่ service
  • The proposed location is a public network drive (e.g. P:\obsidiandev) — readable by anyone on the share.ตำแหน่งที่เสนอคือ network drive สาธารณะ (เช่น P:\obsidiandev) — ใครที่เข้าถึง share ก็อ่านได้
  • Obsidian officially advises against vaults on network/cloud drives — the file watcher and metadata cache corrupt under network latency, and .obsidian/ workspace config collides between concurrent users.Obsidian แนะนำอย่างเป็นทางการว่าไม่ควรวาง vault บน network/cloud drive — ตัวเฝ้าไฟล์และ cache ของ metadata เสียหายภายใต้ความหน่วงของเครือข่าย และ config .obsidian/ ชนกันเมื่อผู้ใช้หลายคนเปิดพร้อมกัน
  • Cross-project reuse — not just capture — is the real goal; without structure the vault becomes write-only.การนำกลับมาใช้ข้ามโปรเจกต์ ไม่ใช่แค่การบันทึก คือเป้าหมายจริง หากไม่มีโครงสร้าง vault จะกลายเป็นที่เขียนอย่างเดียว
2

Decision Driversปัจจัยในการตัดสินใจ

  • Reuse existing capture points and the existing note format — connect pieces, do not build a parallel system.นำจุดจับความรู้และรูปแบบโน้ตที่มีอยู่กลับมาใช้ — เชื่อมชิ้นส่วนเดิม ไม่สร้างระบบคู่ขนาน
  • Make cross-project recall work, not just capture.ทำให้การค้นกลับข้ามโปรเจกต์ใช้งานได้จริง ไม่ใช่แค่บันทึก
  • Never leak secrets/PII onto a shared drive.ห้ามรั่วไหลความลับ/ข้อมูลส่วนบุคคลลงไดรฟ์ที่แชร์
  • Preserve DevStarter's no-build, Markdown-only model.รักษาโมเดล Markdown ล้วนแบบไม่ build ของ DevStarter

Decisionการตัดสินใจ

✅ Markdown emitter (A) into a Git-backed vault (C), with sanitize-on-write and structured recall frontmatterตัว emit Markdown (A) ลง vault ที่หนุนด้วย Git (C) พร้อมล้างข้อมูลตอนเขียนและ frontmatter ที่มีโครงสร้างสำหรับค้นกลับ

DevStarter writes Markdown notes (YAML frontmatter + [[wikilink]]) into a configured Obsidian vault from three capture points — /devstarter-postmortem, /devstarter-bug-postmortem, and a new /devstarter-knowledge command for ad-hoc technique notes. The config template defaults the vault transport to git (network drive supported as a documented fallback with a safe config). Every note is scrubbed inline against a secret/PII deny-list before it is written — the redaction is done by the emit step itself (the pattern set is the one documented by @devstarter-opensource-sanitizer, which can also verify the vault afterward), so secrets/PII never reach the shared vault. A fixed frontmatter schema (tags, language, framework, symptom, root_cause_category, project) makes the vault queryable across projects via Obsidian search / Dataview, and /devstarter-debug greps the vault at session start to surface prior matching root causes — closing the capture-to-reuse loop.DevStarter เขียนโน้ต Markdown (YAML frontmatter + [[wikilink]]) ลง Obsidian vault ที่ตั้งค่าไว้ จากสามจุดจับความรู้ — /devstarter-postmortem, /devstarter-bug-postmortem และคำสั่งใหม่ /devstarter-knowledge สำหรับโน้ตเทคนิคเฉพาะกิจ ค่าเริ่มต้นของการขนส่ง vault ในเทมเพลตคือ git (รองรับ network drive เป็นทางเลือกสำรองที่มีการตั้งค่าปลอดภัยกำกับไว้) ทุกโน้ตถูกล้างข้อมูลแบบ inline ตาม deny-list ของความลับ/ข้อมูลส่วนบุคคลก่อนเขียน — ตัว emit ทำการลบเอง (ชุดรูปแบบเป็นชุดเดียวกับที่ @devstarter-opensource-sanitizer ระบุไว้ และใช้ตรวจสอบ vault ภายหลังได้) เพื่อไม่ให้ความลับ/ข้อมูลส่วนบุคคลไปถึง vault ที่แชร์ schema ของ frontmatter ที่ตายตัว (tags, language, framework, symptom, root_cause_category, project) ทำให้ค้น vault ข้ามโปรเจกต์ได้ผ่าน Obsidian search / Dataview และ /devstarter-debug จะ grep vault ตอนเริ่ม session เพื่อหยิบต้นตอปัญหาเดิมที่ตรงกันขึ้นมา — ปิดวงจรจากการบันทึกสู่การนำกลับมาใช้

Consequencesผลที่ตามมา

Positiveด้านบวก

  • Engineering knowledge accumulates in one searchable place and is reusable across every project.ความรู้เชิงวิศวกรรมสะสมในที่เดียวที่ค้นได้ และนำกลับมาใช้ได้ทุกโปรเจกต์
  • Small lift — reuses existing capture skills, note format, and the sanitizer agent.งานเพิ่มไม่มาก — ใช้สกิลจับความรู้ รูปแบบโน้ต และ agent ล้างข้อมูลที่มีอยู่
  • No-build, Markdown-only identity preserved; opt-in per project via config.คงอัตลักษณ์ Markdown ล้วนแบบไม่ build; เปิดใช้ราย-โปรเจกต์ผ่าน config

Trade-offsสิ่งที่ต้องแลก

  • Git-backed transport asks each user to clone the vault + install the Obsidian Git plugin (one-time).การขนส่งแบบ Git ขอให้ผู้ใช้แต่ละคน clone vault + ติดตั้งปลั๊กอิน Obsidian Git (ครั้งเดียว)
  • Capture is one-way for now; live query (REST/MCP) is deferred.การจับความรู้เป็นทางเดียวในตอนนี้; การค้นแบบสด (REST/MCP) เลื่อนไว้ก่อน

Risks & Mitigationความเสี่ยงและการลด

⚠ Security — secret leakage onto a shared driveความปลอดภัย — ความลับรั่วลงไดรฟ์ที่แชร์

Raw bug notes can carry stack traces, internal URLs, credentials, customer data, and unpatched-vuln details. On a public share these are readable by anyone. Mitigation: scrub-on-write is mandatory and on by default — the emit step redacts every note inline against a secret/PII deny-list before it is written (verifiable afterward with @devstarter-opensource-sanitizer); if obsidian.sanitize is disabled, the emitter refuses to write to a network-transport vault.โน้ตบั๊กดิบอาจมี stack trace, URL ภายใน, credential, ข้อมูลลูกค้า และรายละเอียดช่องโหว่ที่ยังไม่ได้แพตช์ บน share สาธารณะใครก็อ่านได้ การลด: การล้างข้อมูลตอนเขียนเป็นสิ่งบังคับและเปิดโดยค่าเริ่มต้น — ตัว emit จะลบข้อมูลในโน้ตแบบ inline ตาม deny-list ของความลับ/ข้อมูลส่วนบุคคลก่อนเขียน (ตรวจสอบภายหลังได้ด้วย @devstarter-opensource-sanitizer); หากปิด obsidian.sanitize ตัว emit จะปฏิเสธการเขียนลง vault แบบ network

Risk: network-drive cache corruption / .obsidian/ collisions → Mitigation: default to git transport; for the network fallback, the emitter writes only unique-named files (never edits a shared note) and the guide instructs each user to keep .obsidian/ local, off the share. Risk: write-only graveyard → Mitigation: enforced frontmatter schema + the /devstarter-debug recall grep.ความเสี่ยง: cache ของ network drive เสียหาย / .obsidian/ ชนกัน → การลด: ตั้งค่าเริ่มต้นเป็น git; สำหรับทางสำรองแบบ network ตัว emit เขียนเฉพาะไฟล์ชื่อไม่ซ้ำ (ไม่แก้โน้ตที่แชร์) และคู่มือสั่งให้ผู้ใช้เก็บ .obsidian/ ไว้ในเครื่อง ไม่อยู่บน share ความเสี่ยง: สุสานที่เขียนอย่างเดียว → การลด: บังคับ schema ของ frontmatter + การ grep ค้นกลับใน /devstarter-debug
