Contextบริบท

DevStarter wants to offer codebase-analysis capabilities — knowledge graph, visual dashboard, Q&A, change-impact, onboarding, domain extraction, and wiki analysis. A mature open-source tool, Understand-Anything (MIT, Lum1104), already provides all of these. Critically, Understand-Anything is itself a Claude Code plugin that installs into the same ~/.claude/ home as DevStarter, so the two already coexist with no integration work. The question is how to surface its 8 commands inside the DevStarter workflow.DevStarter ต้องการให้บริการความสามารถวิเคราะห์โค้ดเบส — knowledge graph, dashboard แบบเห็นภาพ, การถาม-ตอบ, การวิเคราะห์ผลกระทบ, onboarding, การสกัด domain และการวิเคราะห์ wiki เครื่องมือโอเพนซอร์สที่พัฒนาแล้วอย่าง Understand-Anything (MIT, Lum1104) มีครบทั้งหมดนี้อยู่แล้ว ที่สำคัญคือ Understand-Anything เองเป็นปลั๊กอินของ Claude Code ที่ติดตั้งลงใน home ~/.claude/ เดียวกับ DevStarter ทั้งสองจึงอยู่ร่วมกันได้โดยไม่ต้องทำงานผสานรวมใด ๆ คำถามคือจะนำคำสั่งทั้ง 8 ตัวมาแสดงภายใน workflow ของ DevStarter อย่างไร

📎 Constraintsข้อจำกัด
  • DevStarter's identity is pure Markdown + Bash — no build step, no runtime, installed by copy.อัตลักษณ์ของ DevStarter คือ Markdown + Bash ล้วน — ไม่มีขั้นตอน build ไม่มี runtime ติดตั้งด้วยการคัดลอก
  • Understand-Anything is a TypeScript monorepo needing pnpm build + tree-sitter WASM; the dashboard needs Node.js.Understand-Anything เป็น TypeScript monorepo ที่ต้อง build ด้วย pnpm และใช้ tree-sitter WASM ส่วน dashboard ต้องใช้ Node.js
  • All DevStarter commands must keep the devstarter- prefix.คำสั่งทุกตัวของ DevStarter ต้องคงคำนำหน้า devstarter-
  • Understand-Anything is MIT-licensed — attribution required if vendored.Understand-Anything ใช้สัญญาอนุญาต MIT — ต้องระบุที่มาหากนำซอร์สเข้ามา
2

Decision Driversปัจจัยในการตัดสินใจ

  • Preserve DevStarter's no-build, Markdown-only model.รักษาโมเดล Markdown ล้วนแบบไม่ build ของ DevStarter
  • Deliver the full capability set with minimal effort and maintenance.ส่งมอบความสามารถครบชุดด้วยความพยายามและการดูแลที่น้อยที่สุด
  • Avoid drift from an actively developed upstream.หลีกเลี่ยงการ drift จากต้นทางที่กำลังพัฒนาอยู่
  • Keep licensing and attribution obligations simple.ทำให้ภาระเรื่องสัญญาอนุญาตและการระบุที่มาเรียบง่าย

4

Decisionการตัดสินใจ

✅ We will adopt Option A — thin wrappersเราจะเลือกทางเลือก A — ตัวห่อหุ้มแบบบาง

We will add 8 devstarter-understand* commands (SKILL.md + SDLC runbook each) that detect the Understand-Anything plugin and delegate to its matching command, passing arguments through. Because the plugin already coexists in ~/.claude/, this delivers the entire capability set while keeping DevStarter pure Markdown — no build, no vendored code, no naming exception. Upstream improvements flow through automatically.เราจะเพิ่มคำสั่ง devstarter-understand* จำนวน 8 ตัว (อย่างละ SKILL.md + SDLC runbook) ที่ตรวจหาปลั๊กอิน Understand-Anything แล้วส่งต่อไปยังคำสั่งที่ตรงกันพร้อมส่ง argument ต่อ เนื่องจากปลั๊กอินอยู่ร่วมกันใน ~/.claude/ อยู่แล้ว วิธีนี้ส่งมอบความสามารถครบชุดพร้อมคง DevStarter ให้เป็น Markdown ล้วน — ไม่ต้อง build ไม่มีโค้ดที่ vendor ไม่ต้องยกเว้นการตั้งชื่อ และการพัฒนาจากต้นทางไหลผ่านมาอัตโนมัติ

Consequencesผลที่ตามมา

Positiveด้านบวก

  • Full codebase-analysis suite available as native DevStarter commands at near-zero cost.ชุดวิเคราะห์โค้ดเบสครบใช้งานเป็นคำสั่ง DevStarter ในตัวด้วยต้นทุนแทบเป็นศูนย์
  • DevStarter's no-build identity is preserved.คงอัตลักษณ์แบบไม่ build ของ DevStarter ไว้
  • Upstream updates require no DevStarter changes.อัปเดตจากต้นทางไม่ต้องแก้ DevStarter

Trade-offsสิ่งที่ต้องแลก

  • Runtime dependency: the plugin must be installed. Mitigated by a preflight detect-and-prompt in every wrapper.dependency ตอนรันไทม์: ต้องติดตั้งปลั๊กอิน แก้ด้วย preflight ตรวจและแจ้งเตือนในตัวห่อหุ้มทุกตัว
  • The dashboard command needs Node.js — a requirement owned by the plugin, not DevStarter.คำสั่ง dashboard ต้องใช้ Node.js — เป็นข้อกำหนดของปลั๊กอิน ไม่ใช่ DevStarter

Risks & Mitigationความเสี่ยงและการลด

Risk: upstream renames or removes a command → Mitigation: each wrapper is a single delegating line, trivial to update. Risk: naming confusion with the existing /devstarter-onboard → Mitigation: the prefix keeps devstarter-understand-onboard distinct; both runbooks note the difference.ความเสี่ยง: ต้นทางเปลี่ยนชื่อหรือลบคำสั่ง → การลด: ตัวห่อหุ้มแต่ละตัวเป็นบรรทัดส่งต่อเดียว แก้ไขง่าย ความเสี่ยง: สับสนชื่อกับ /devstarter-onboard เดิม → การลด: คำนำหน้าทำให้ devstarter-understand-onboard แยกกัน และ runbook ทั้งสองระบุความต่างไว้
